# Speaker Script

This script assumes a 20-minute presentation with 2 to 3 presenters.

## 0:00 to 1:30 - Opening

Say:

"Southlake's challenge is not just a data problem. It is a safe experimentation problem. Healthcare teams need to model new care pathways, but they cannot casually use real patient records for early testing. We built a synthetic data studio that creates a safer planning sandbox."

## 1:30 to 4:00 - Why This Matters

Say:

"Southlake is moving toward a distributed health network. That means thinking beyond a single hospital building and asking where care should happen, what might move closer to home, how advanced-care-campus capacity could change, and how routing should evolve across observation, transfer, and community-connected pathways. Those questions are data-heavy, but privacy and governance make experimentation hard. Synthetic data can unlock earlier-stage innovation."

## 4:00 to 10:00 - Live Demo

Say:

"Here is the studio. We start from a curated public emergency-department dataset, so the workflow is safe to demonstrate."

Then do the live demo:

- Click `Load recommended demo run`
- Point to the Southlake-specific narrative cards at the top of the screen
- Show the saved `Distributed Campus Routing` output first
- If time allows, scroll back to the builder and explain that the same studio can also generate a fresh run

While the run is processing, say:

"The reason we call this agentic is that the system does more than generate rows. It frames the planning intent, profiles the data, creates a synthesis plan, applies a Southlake-style scenario, evaluates the result, and writes caution notes."

When results appear, say:

"Now we can see the output metrics, the source-versus-synthetic preview, and the pitch-ready summary. The studio is designed to help an innovation team move from idea to a testable planning artifact quickly, without pretending this is already a clinical system."

## 10:00 to 13:00 - Slide 1 Methodology

Say:

"Our methodology is a five-step flow. We load a public dataset, frame the planning goal, generate synthetic data, evaluate the result, and export planning artifacts. That structure matters because in healthcare, the explanation and caution layer is just as important as the data itself."

## 13:00 to 16:00 - Slide 2 Features

Say:

"The most important product features are not only generation quality. They are Southlake-specific scenario control, visible evaluation, saved demo reliability, and the ability to turn a run directly into presentable materials. That is what makes this a practical hackathon prototype rather than a hidden backend model."

## 16:00 to 18:00 - Slide 3 Cautions

Say:

"We want to be very clear about limitations. This is for planning and prototyping, not clinical action. The source is a U.S. public-use dataset, not Southlake's real data. Synthetic outputs still need governance and validation before any operational use. We see this as a safer front-end to innovation, not the final decision system."

## 18:00 to 20:00 - Closing

Say:

"Our core argument is simple: agentic synthetic data can help health systems test future-state ideas earlier, faster, and more safely. For Southlake, that means a practical first step toward distributed-network experimentation without touching real patient records."

End with:

"If this moved beyond the hackathon, the next step would be validation with governed local data, privacy review, and scenario designs tailored to Southlake's actual operating questions."
