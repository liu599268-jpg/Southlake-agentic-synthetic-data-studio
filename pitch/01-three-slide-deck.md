# Three-Slide Deck

This is the final structure for the required 3-slide presentation deck.

## Slide 1: Methodology

### Title

From Public ED Data To Southlake Future-State Planning Scenarios

### Core Message

We built an agentic workflow that turns safe public ED data into synthetic planning artifacts for Southlake's distributed-health-network questions, then makes the limitations visible before anyone over-trusts the output.

### On-Slide Content

- Input: curated public emergency-department dataset used as a safe planning stand-in
- Agent workflow: goal framing -> data profiling -> synthesis planning -> generation -> evaluation -> caution reporting
- Southlake-aligned scenario layer: distributed campus routing, community diversion, regional growth, ED surge
- Output: synthetic dataset, metrics dashboard, caution summary, export ZIP, pitch artifacts

### Visual Recommendation

Use `assets/architecture-workflow.svg` as the main visual.

### Speaker Notes

- Start by saying the problem is not just generating fake rows.
- The real challenge is generating useful synthetic planning data for Southlake's future-state questions and making the limitations visible.
- Emphasize that the system behaves like an agent because it reasons across multiple steps.
- Mention that the future-state story is about routing care across campus, observation, transfer, and community-connected settings.

## Slide 2: Features

### Title

What The Studio Actually Does

### Core Message

The product is not only a data generator. It is a planning and innovation studio with scenario controls, saved demo runs, evaluation, and exportable outputs.

### On-Slide Content

- Southlake-specific scenario presets aligned to distributed-network planning needs
- One-click saved demo runs plus optional fresh synthesis or CSV upload
- Synthetic output preview with fidelity, privacy, and utility metrics
- Agent timeline explaining what the system did
- Auto-generated methodology, features, and cautions for pitch preparation

### Visual Recommendation

Use a screenshot of the run page from `assets/full-run-page.png` with 5 callout labels:

- Saved demo-run shortcut
- Scenario controls
- Metrics
- Source vs synthetic preview
- Pitch/caution output

### Speaker Notes

- Point out that judges can see both the data product and the explanation product.
- Stress that this makes the tool useful for innovation teams, not just technical users.
- Mention that the app now includes a reliable saved-run path, so the presentation does not depend on live generation speed.

## Slide 3: Cautions

### Title

What The Studio Is Safe For, And What It Is Not

### Core Message

Synthetic data is useful for planning and experimentation, but it is not local operational truth and should not be used without governance and validation.

### On-Slide Content

- For planning, innovation, and prototyping, not direct clinical or operational decisions
- U.S. public-use source does not equal Southlake's real patient mix
- Single-table prototype only; not a full hospital digital twin
- Synthetic outputs still need privacy, quality, and governance review before real-world use
- Best next step is validation with governed local data under proper controls

### Visual Recommendation

Use a simple two-column layout:

- Left column: `Good For`
- Right column: `Not Ready For`

### Speaker Notes

- This slide increases credibility.
- Show that the team understands healthcare risk and is not overselling the prototype.
- Judges are more likely to trust a careful team than a team that claims too much.
